This guide is for developers integrating with the Slimecoin protocol.

## Minimum player setup

Before a player can enter a game, an app should ensure:

1. The player has a vault `Profile`.
2. The player has a regional `PlayerPayouts` account.
3. The profile session authority is the signer that will submit gameplay transactions.
4. The selected validator matches the player's region.
5. The player has enough USDC, SLIME, or a valid free queue entry.

## Starting a game

The game program handles buy-in reservation and matchmaking. The app must pass the correct accounts:

- player wallet/profile/payouts
- validator and rollup ledger
- queue PDA for game, mint, validator, and buy-in
- game session PDA
- game signer PDA
- vault program
- matchmaking program
- MagicBlock accounts for delegated sessions and cranks where required

For shared-seed v3 games, the join instruction may return one of two practical states:

| State | Meaning |
| --- | --- |
| Matched | The session already has an opponent and seed; the game is head-to-head |
| Placeholder | The player's run is waiting for a future opponent using the same seed path |

## Submitting scores

For score-after-play games, the game submits a final score into matchmaking. Matchmaking either queues it or immediately settles it against an opponent.

For shared-seed games:

- unmatched players submit a score with the seed
- matched players use the payout/resolve path
- timeout cranks can submit forfeits or resolve stale matches

Apps should treat "game over" state as authoritative only after reading the session account. UI timers are helpful, but the session's `game_end_slot`, `game_over`, score, and run ID are the settlement inputs.

## Claiming rewards

After settlement, balances may still be in `PlayerPayouts`:

```text
claimable_usd
claimable_slime
claimable_slimecoin
tournament_payout
weekly_leagues_payout
```

Apps should offer claim actions that move these amounts into the profile. Do not assume that a settled match immediately changes `Profile.usd_balance`.

## Queue derivation

Queues are tied to:

- game ID
- validator
- mint
- buy-in

Shared-seed v3 queues use the numeric `u32` game ID encoded into a 32-byte `game_id` field. Avoid inventing alternate queue names unless the protocol source explicitly supports them for that game version.

## Handling stale sessions

A player may have a previous active session. Correct client behavior is:

1. Fetch the session PDA.
2. If it is active and owned by the same player, resume it.
3. If it is finished, submit or resolve the score before starting a new run.
4. If it is reset or empty, start a new run.

This preserves locked buy-ins and prevents duplicate active sessions for the same player/game/validator.
